# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

A Vite + React 19 + TypeScript project whose **product is a single reusable component: `DataGrid`** (built on TanStack Table v8). The retail "Recommendation Workbench" screen is *only a demo consumer* that exercises the component's public surface — it is not the deliverable.

`AGENTS.md` holds the detailed contribution rules (directory layout, naming, the component-surface boundary). This file does not repeat them; it focuses on commands and the cross-file architecture you only get by reading `DataGrid.tsx` end to end.

## Commands

- `npm run dev` — Vite dev server at http://127.0.0.1:5173/ (host pinned to 127.0.0.1).
- `npm run build` — `tsc -b && vite build`. The `tsc -b` step **is** the type-check (project references in `tsconfig.json` → `tsconfig.app.json` for `src`, `tsconfig.node.json` for config files). Run `npx tsc -b` alone for a type-check without bundling. This builds the **demo app**, not the library.
- `npm run build:package` — build the distributable **library** to `dist/`: ESM `datagrid.js` + CJS `datagrid.cjs`, both with a `"use client"` banner; rolled-up `datagrid.d.ts` + `datagrid.d.cts` declarations from `vite-plugin-dts`/API Extractor; and minified, hand-written `datagrid.css` from `src/components/DataGrid/datagrid.css` via esbuild. This script runs `clean:dist` first so stale demo-app files cannot leak into the tarball. `package.json` exposes conditional import/require types and runtime files plus `./styles.css` and `./package.json` subpaths.
- `npm run test:package` — run the package build, pack it, assert the exact tarball surface, smoke-test ESM/CJS runtime imports and the CSS export, compile NodeNext ESM/CJS consumer type fixtures against the extracted package, then gate the artifact with publint and Are the Types Wrong. Keep this serialized with any other command that mutates `dist/`.
- `npm run preview` — serve the production `dist/` build.
- `npm test` — `vitest run` (single pass, no watch; jsdom env from `vite.config.ts`).
  - Single file: `npx vitest run src/components/DataGrid/DataGrid.test.tsx`
  - Single test by name: `npx vitest run -t "uses column visibility"`
  - Watch mode: `npx vitest`
  - Tests are split by concern next to the engine: `DataGrid.test.tsx` (pivot) plus `DataGrid.{grid,a11y,usability,controlled,config,filters,virtual}.test.tsx`, and `src/utils/formatters.test.ts`. Behavioral work here is test-first.

There is **no ESLint/Prettier setup** — don't invent a `lint` script. Type safety is enforced only by `tsc -b`.

## The prime architectural rule

`src/components/DataGrid/` must stay **domain-neutral and generic over `TData extends object`**. No retail field names, copy, status values, labels, or mock-data assumptions belong inside it. Anything domain-specific lives in `src/data/` (configs + mock rows), `src/demo/` (detail panels, actions), or `src/App.tsx` (composition). New configurable behavior is added as a typed prop or a feature flag, never a hardcoded assumption. See `AGENTS.md` § "Component-Surface First" for the full policy.

## Architecture (the engine is `DataGrid.tsx`; state + pure logic are extracted into siblings)

`DataGrid.tsx` (~2.3k lines) is the orchestration engine — it composes state, builds column defs, owns the single `useReactTable` call, and renders the shared table chrome. It runs top-to-bottom through 11 numbered phases, each marked with a `// ----- N. Phase -----` banner (a matching table-of-contents comment sits just above the `DataGrid` function) — read those banners to navigate it rather than scrolling. The heavy logic is pulled into focused siblings:

- **State & helper hooks:** `useGridState.ts` owns the controlled/uncontrolled hybrid state triad for every slice (see the bullet below); `useCellFocus.ts` owns the roving-`tabIndex` geometry + focus; `useCellEditing.ts` owns the inline-edit state machine. `gridHelpers.ts` holds the pure utilities they share (`resolveUpdater`, `normalizeColumnPinning`, `flattenExpandedRows`, `collectExpandableGroupIds`, `getSelectionStatus`, the `isPivotRow`/`isGeneratedPivotColumnId` guards); `columnVirtual.ts` holds the pure partition/window/row-assembly helpers for horizontal column windowing.
- **Cells & filtering:** `cells.tsx` (the value-erased `AnyColumnConfig` + `renderCellValue`/`getColumnSearchText`/`getCellClasses`/`renderGroupingValue`), `filterDefaults.ts` (pure `dataType`→`GridFilterType` inference: `defaultFilterTypeForDataType`/`resolveFilterType`, shared by the grid and the server-query paths), `filterMatch.ts` (the unified `matchesFilterValue` predicate + `dateSortingFn`), `filters.tsx` (filter-control *chrome*: header popover, floating row, pivot Filters popover; exports `summarizeFilter`/`isFilterActive`) delegating to `filterBodies.tsx` (the per-type control bodies + `formatOptionLabel`), `AppliedFilters.tsx` (the applied-filter chip bar), `cellEditor.tsx` (inline edit UI + parse/validate).
- **Pivot:** `pivot.tsx` owns materialization, with pure helpers split into `pivotColumns.ts` (column-bucket tree, value computation) and `pivotHelpers.ts` (stable ID builders `groupId`/`measureColumnId`/`leafId`, source-value access, measure aggregation).
- **Columns / storage / export:** `columnGroups.ts` (`buildGroupedColumnDefs` + the `DataGridColumnGroup` type), `storage.ts` (scoped `localStorage` helpers), `src/utils/export.ts` (CSV/TSV + clipboard).
- **Toolbar:** `Toolbar.tsx` is a dumb prop-driven shell that arranges the sub-controls, each its own component: `ToolbarSearch.tsx`, `ToolbarColumns.tsx` (column menu + drag reorder), `ToolbarGrouping.tsx` (group-by builder), `ToolbarSavedViews.tsx`, plus the Export CSV button and the pivot Filters popover. `icons.tsx` and `trendIconSet.tsx` hold local Lucide-derived SVG components; there is no `lucide-react` runtime dependency.

The published package has no regular runtime `dependencies`. React, React DOM, and TanStack Table are peer dependencies and remain external to the bundles. `@tanstack/react-virtual` is an internal implementation detail bundled into the library; it stays a dev dependency only. The package's styling source is `src/components/DataGrid/datagrid.css`: it is scoped to `dg-*` classes, uses CSS custom properties for theming, and owns the flash/sheet keyframes shared by the package and demo.

`index.ts` is the public boundary — **export any new public type there**. Subsystems to understand before editing:

- **Controlled / uncontrolled hybrid state (in `useGridState.ts`).** Every state slice (sorting, globalFilter, columnFilters, columnVisibility, columnSizing, columnOrder, columnPinning, pagination, rowSelection, grouping, expanded, pivot, savedViews, activeViewName) follows one pattern: a `current*` value resolves as `controlledState?.X ?? internalUseState`, and an `emit*Change` helper resolves the updater, writes internal state **only when that slice is uncontrolled** (`controlledState?.X === undefined`), persists if needed, then fires the optional `on*Change` callback. `useGridState` returns the `current*` values + `emit*Change` helpers; `DataGrid.tsx` consumes them. Each slice is independently controllable. When adding state, replicate this triad (`current*`, `emit*Change`, optional `on*Change` prop) inside `useGridState` — do not call `setState` directly in handlers.

- **Client vs server data (`dataMode`).** `dataMode="server"` (grid layout only; ignored in pivot) flips `useReactTable` into `manualSorting/Filtering/Pagination` and trusts the supplied `data` (current page) + `rowCount`. It rides the controlled-state triad — the existing `on*Change` callbacks are the server query. Degradations are centralized in a `dataModeFeatureDefaults` merge (parallel to `layoutFeatureDefaults`): grid grouping and summaries default **off** (can't be correct over one page), filter `select` options come only from static `GridFilterConfig.options`, and export/select-all/`aria-rowcount` scope to the loaded page (with `aria-rowcount` reflecting `rowCount`). All defaults stay consumer-overridable via `features`. The demo's `src/data/fakeServer.ts` simulates the backend.

- **Feature flags × layout modes.** `features` is merged as `{ ...defaultFeatures, ...layoutFeatureDefaults, ...dataModeFeatureDefaults, ...featureOverrides }` (see the dataMode bullet above for the server-mode defaults). `layoutMode` (`"grid"` | `"pivot"`) keeps pivot grouping controls on, but pivot rows/measure columns are materialized before `useReactTable` and then rendered through the shared `table.getHeaderGroups()` / `row.getVisibleCells()` path. Pivot row-label, measure, and column-axis bucket columns are generated `ColumnDef`s, so sorting, resizing, visibility, ordering, pinning, saved views, pagination, and virtualization use the standard table chrome.

- **Persistence is opt-in and scoped.** Only when a `storageKey` prop is given are exactly four slices written to `localStorage`: `${storageKey}.columnSizing`, `.columnOrder`, `.columnPinning`, `.savedViews`. All other state is in-memory. Writes live **inside** the `controlledState?.X === undefined` guard, so a slice the consumer controls is never persisted by the grid. The `loadJson`/`saveJson`/`removeJson` helpers guard `typeof window === "undefined"` and swallow parse errors. Never introduce an unscoped/global key (two grids would collide).

- **Column config drives cell rendering.** `GridColumnConfig<TData>` (in `src/types/grid.ts`) is the public column contract — a **per-key distributive union** so value-typed callbacks (`formatValue`, `getCellClassName`, `conditionalFormats`, `validate`, `parseValue`, `renderEditCell`…) receive `TData[accessorKey]`, not `unknown`. Internally the engine casts columns once to a value-erased `AnyColumnConfig<TData>` (in `cells.tsx`). `renderCellValue` dispatches on `column.dataType` (`text` | `number` | `currency` | `percent` | `status` | `date`) to the locale-aware formatters (driven by `locale`/`currency`/`dateFormat` props), unless `column.formatValue` overrides; non-finite/blank numerics and unparseable dates render empty (no `$NaN`/fake `$0`). `date` columns keep `accessorKey` (so typed callbacks get the raw value, not a normalized `Date`) and sort via `dateSortingFn` (chronological across `Date`/ISO/epoch; blanks last ascending). `status` pills are styled by `getStatusClassName` **or** the declarative `statusStyles` map. Numeric types right-align with `tabular-nums`; per-cell classes come from `getCellClassName` **and** declarative `conditionalFormats` (composed).

- **Value-driven cell visual effects (grid mode only).** Five declarative props on `GridColumnConfig` add *continuous* visualization beyond the *binary* `conditionalFormats`: `colorScale` (background heatmap, 2-pt or 3-pt diverging), `dataBar` (in-cell bar, zero-baseline for domains crossing zero), `iconSet` (consumer-supplied icon by predicate rule; `trendIconSet` uses the local SVG set), `progressBar` (renders a `percent` cell as a 0–100% bar), `flashOnChange` (green-up/red-down on value change). Pure math lives in `cellEffects.ts` (color interpolation, bar geometry, domain compute), React pieces in `cellEffectsRender.tsx`. **Content** effects (`iconSet`, `progressBar`) render inside `renderCellValue`; **box** effects (`colorScale` bg + readable text, `dataBar` fill, flash overlay) are applied at the `<td>` in `DataGrid.tsx`. `colorScale`/`dataBar` need a per-column numeric **domain** (min/max over the **filtered** rows; page-scoped in server mode) computed in a `columnDomains` memo; pass an explicit `domain` to pin it. **Flash** is virtualization-safe: a grid-level previous-value map keyed `rowId:columnId`, diffed in a `useEffect` on `data` (seeded silently on mount, entries cleared after `duration`), so cells don't false-flash on scroll. Effect classNames (`conditionalFormats`/`getCellClassName`) win over the `colorScale` inline background. Flash keyframes + the `prefers-reduced-motion` guard live only in `datagrid.css`. Pivot measure columns are out of scope.

- **Mobile card layout (grid mode, opt-in).** `features.cardLayout` + the `cardView` prop switch the render path to auto-composed cards below a container-width breakpoint (default 640px, measured on the grid root via `useContainerWidth.ts` — container-driven, not viewport). `cardComposition.ts` (pure) assigns every visible leaf column a role (title/badge/subtitle/metrics/meta) by dataType + visible order; `DataGridCardConfig` overrides any role; `renderCard` replaces the body. `CardList.tsx` renders list semantics through the same table instance (`renderCellValue` formatting, colorScale/dataBar carried onto metric tiles); `ToolbarCompact.tsx` + `BottomSheet.tsx` provide the search/sort/filters chrome writing the existing `sorting`/`columnFilters` slices (`FilterBody` reuse). Card mode defaults desktop-only features off via `cardModeFeatureDefaults` (same merge mechanism as server mode; `headerFilters` stays ON — it feeds `resolvedFilters` → the Filters sheet). Pivot ignores card mode. `cardView.mode` pins the mode (how jsdom tests force it); sheet keyframes live in `datagrid.css`.

- **Filtering is auto-provisioned and type-aware.** Every non-synthetic leaf column is filterable by default with a control inferred from its `dataType` (`filterDefaults.ts`: `defaultFilterTypeForDataType` + cardinality-aware `resolveFilterType` — low-cardinality `text` auto-facets to `multiSelect`; `status`→`multiSelect`; numerics→`range`; `date`→`date`; `boolean`→`boolean`). A single **`resolvedFilters`** memo in `DataGrid.tsx` merges that inference with optional `filters` *overrides* and is the one source every consumer reads — the `filterTypeByColumnId`/`filterOperatorByColumnId` maps, the grid header funnels, the pivot toolbar **Filters** popover, and the applied-filter chip bar — so grid and pivot stay in lockstep by construction. Flags: `features.headerFilters` (master toggle), `autoColumnFilters` (false ⇒ only columns named in `filters` are filterable — legacy opt-in), `filterSummary` (the chip bar); the `facetThreshold` prop tunes the auto-facet cardinality cutoff; `GridColumnConfig.enableFiltering:false` is a **hard** per-column opt-out (wins even if the column is named in `filters`) and `GridFilterConfig.filterable:false` opts a named column out. The **`AppliedFilters`** chip bar (`AppliedFilters.tsx`, gated by `features.filterSummary`) renders a removable chip per active filter + a global-search chip + Clear all, reusing `summarizeFilter`/`isFilterActive` from `filters.tsx`. Grid renders per-leaf-column header popovers (plus an opt-in floating row, `features.floatingFilters`); pivot renders the consolidated toolbar popover. Both write the existing `columnFilters` slice. One predicate, `matchesFilterValue` (in `filterMatch.ts`), supports `select` (exact), `multiSelect` (array), `range` ({min,max} + `gt`/`gte`/`lt`/`lte`/`equals`/`notEquals`), `text` (case-insensitive contains vs **formatted** text), `date` ({from,to}, `toDate`-normalized, checked **before** the range branch), and `boolean` (`toBoolean`, true/false). It is threaded into **both** filter paths — the grid `columnFilterFn` closure **and** the pivot `pivotSourceRows` loop — which must stay in lockstep (they read the maps; this feature only changed how the maps are *built*). Global search matches **formatted** text via `getColumnSearchText`. **Server parity:** `fakeServer.ts` and the `serverQuery.ts` `requestToQuerySpec` helper both derive filter types from the *same* `resolveFilterType({ isServerMode: true })` (overrides win) — keep any new server path on this shared inference so it stays in lockstep with the grid. **Degradations (by design):** in server mode (or with `autoColumnFilters` off) a `text`/`status`/categorical column with no static `options` becomes an empty `multiSelect`/free-text box — supply `options` for categoricals used in server mode (the demo does for `department`/`recommendation_status`); and because the `text` control is data-dependent (distinct count vs `facetThreshold`), it can flip `multiSelect`↔contains as client data changes or between client/server mode. An empty option list renders an explanatory "No options available" message instead of a blank popover, and a `multiSelect` body with more than 10 options gets an in-list search box that narrows the displayed rows only — selection state is untouched, so scrolled-out selections stay selected.

- **Keyboard cell navigation & editing.** Geometry/focus lives in `useCellFocus.ts`, the edit state machine in `useCellEditing.ts`, and `onCellKeyDown` (phase 9 in `DataGrid.tsx`) is the precedence-chain orchestrator wiring them to row actions and clipboard. Cells own focus via roving `tabIndex` over a rectangular grid (leaf data rows × non-select visible leaf columns; group rows keep their own focusable toggle). `focusedCell` and `editingCell` are internal state (the `activeRow` pattern — not controlled slices), with optional `onFocusedCellChange`. Arrows/Home/End/Ctrl+Home/Ctrl+End/PageUp/PageDown move; the precedence chain is Enter/F2 → edit (if `editable`) → row action → group toggle, Space → toggle selection (grid), Escape → cancel edit → close detail panel. Editing is **controlled**: commit (Enter/Tab/blur) runs `parseValue` + `validate` and emits `onCellEdit({rowId,row,columnId,value,previousValue})` — the grid **never mutates `data`**. The editor (`cellEditor.tsx`) stops key propagation so it never double-fires with cell-nav or the document Escape listener. Blur commits when the draft value is valid and reverts (cancels, no `onCellEdit`) when invalid, so a mouse-only user is never trapped in an open editor; Enter still stays open and shows the validation error on an invalid value. `aria-rowcount`/`aria-colcount`/`aria-rowindex`/`aria-colindex` are set (rowindex offset by header rows). When `features.fillHandle` is on (inert unless `cellSelection` is also on), a small handle at the bottom-right corner of the selection/focused cell drags down/right to replicate its value(s) into the cells passed over, and `Ctrl+D`/`Ctrl+R` provide the keyboard equivalent over an existing multi-row/multi-column selection (top row / left column tiled into the rest via modulo indexing). Both paths commit through `commitFill`, the same validated-edit contract as paste — `computeEditError` gates each target (no `parseValue` step, since the source is already-typed data, not clipboard text) — silently skipping non-editable/invalid targets, and leave the union of source + filled cells selected.

- **Export & clipboard.** The toolbar **Export CSV** button (`features.export`) exports the current view (formatted text, visible columns in order) — all filtered rows across pages, or selection-only when rows are selected (cross-page via `getSelectedRowModel`). Synthetic `select` and row-actions columns are excluded. **Ctrl/Cmd-C** (`features.clipboard`) copies the selection or focused cell as TSV; it only fires with a cell focused, so editors / filter inputs keep the native copy. Ctrl/Cmd-Shift-C copies with a header row; `clipboardIncludeHeaders` makes plain copy include headers, and `clipboardValueMode="raw"` copies underlying values (fraction percents, plain numbers, ISO dates, raw status codes; `formatValue` ignored) instead of formatted text. Helpers (`toCsv`/`toTsv`/`downloadTextFile`/`writeClipboardText`) live in `src/utils/export.ts`.

- **Column header groups (grid mode).** `columnGroups` (a `DataGridColumnGroup` tree) is assembled by `buildGroupedColumnDefs` into nested `ColumnDef` bands rendered through the standard `getHeaderGroups()` path (`<th colSpan>` follows visible leaves). Pivot ignores it. Don't split a single `columnGroups` band across pinned and unpinned columns — TanStack's unified `getHeaderGroups()` can fuse adjacent same-group headers across the pin boundary, and the header windowing path assumes one pin region per header instance.

- **Summaries and pivot measures.** `summaryItems` render a summary bar in grid mode and are adapted into pivot measures for compatibility. New pivot consumers should pass `pivot.measures` with built-in or custom aggregations. Each summary item's `value`/`description` receives a `DataGridSummaryContext` (`rows`, `filteredRows`, `selectedRows`, `allRows`, `scope`). `scope` is `"filtered"` | `"selected"` | `"group"`; `summarySelectionMode="auto"` flips filtered→selected once any row is selected.

- **Grouping, expansion & row windowing.** Grid mode uses TanStack's grouped/expanded row models. Pivot mode owns grouping in the materializer via `pivot.rows` and stable IDs such as `pivot:group|department=Grocery`; do not reintroduce a pivot-only table renderer. `pivot.columns` builds nested generated header groups and per-bucket measure IDs like `measure:revenue|col:department=Grocery`, plus subtotal and grand-total generated columns. `totalBehavior` is resolved in the materializer for total buckets. `visibleRows` is the single source of what renders. For grid grouping without pagination, flatten over the nested `getExpandedRowModel()`; grid + pagination and pivot use `table.getRowModel().rows`. Pivot pagination defaults to `topLevelGroups` (manual page count, children stay with parents) and can opt into flat `visibleRows`. When `virtualizeRows` is set, `renderBodyRows` windows `visibleRows` via `@tanstack/react-virtual` using top/bottom spacer `<tr>`s + `measureElement`. When `virtualizeColumns` is set, unpinned center columns are likewise windowed via a horizontal virtualizer (`columnVirtual.ts` holds the pure partition/window/row-assembly helpers); pinned columns always render, header-band `colSpan`s are clipped to rendered leaves, and spacer `<col>`/`<th>`/`<td>`s (`.dg-cell-spacer`) preserve fixed-table geometry; card layout ignores it.

- **Generated pivot column state.** Pivot generated IDs must be reconciled whenever axes/measures change. Keep `pivot:rowLabel` and current `measure:*` IDs, prune stale generated IDs from visibility/order/pinning, and append missing generated IDs in the materialized order.

- **Pivot selection.** Pivot selection defaults to `sourceRows`: selecting an aggregate row toggles every source row under it, header select-all toggles all filtered source rows, and selected counts/summaries read source IDs from `rowSelection`. Other selection modes fall back to pivot-row selection and should be treated as less complete until intentionally expanded.

- **Detail panel & active row.** `activeRow` is internal state (not a controlled slice) surfaced via the `onActiveRowChange` callback. `renderDetailPanel(row, { close })` is handed a close control; clicking a row opens it, clicking the active row again toggles it shut, and Escape closes it (gated so a cell edit wins Escape first). Rows are **not** the tab stop — cells own keyboard focus (see "Keyboard cell navigation & editing"); the row keeps `onClick` for mouse, and Enter on a focused cell triggers the row action.

- **Row actions.** `rowActions` adds a domain-neutral synthetic actions column, gated by `features.rowActions`. It accepts a static action array or per-row resolver; each action may be hidden/disabled per row. The column is right-pinned by default when column pinning is enabled, excluded from export/clipboard, column chooser, and roving cell navigation, and row-action buttons stop row-click/detail propagation.

## Demo / consumer layer

`src/App.tsx` wires it together with a Grid/Pivot layout toggle (pivot = summary subtotals; grid = item-level rows), `columnGroups` header bands, row actions for recommendation status changes, opts into `virtualizeRows`, and holds `data` in state so `onCellEdit` can apply edits by `item_id` (the grid never mutates `data`). The demo keeps the item detail panel off by passing `features={{ detailPanel: false }}`. `src/data/mockRetailData.ts` defines the `RetailItem` type (incl. a `last_restocked_at` date field), the retail column/filter/summary configs (filters use `select` / `multiSelect` / `range` / `text` / `date` types; `units`/`recommendation_status`/`last_restocked_at` are `editable`), status pill styles, and 500 deterministic synthetic rows (`seededRandom` via `Math.sin` — stable across reloads). `src/demo/RetailDetailPanel.tsx` remains demo-only detail-panel UI if a consumer re-enables `renderDetailPanel`. `src/utils/formatters.ts` holds the shared, locale-aware currency/number/percent/signed-percent/status-label/date formatters (`formatDate`/`toDate`) used by both the grid and the demo.
